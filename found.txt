[CRITICAL ISSUES]

1. MULTI-TENANT SECURITY GAP ❌

Current model:
organizationId passed manually.

Risk:
developer forgets → global data exposure.

RULE:
Tenant filtering must be AUTOMATIC.

Fix:

Repository receives context once:

constructor(context)

this.organizationId = context.organizationId

Never pass orgId manually per query.

--------------------------------------------------

2. buildOrgFilter BUG (SERIOUS)

Current:
`${columnName} = `

Relies on external `$index`.

Very fragile.

Return FULL clause:

`${columnName} = $1`

or accept paramIndex.

Broken filters = silent data leaks.

--------------------------------------------------

3. TABLE NAME INJECTION RISK ⚠️

this.tableName interpolated directly.

Safe only if static.

Rule:
Never allow dynamic table names.

Freeze repository tables at construction.

--------------------------------------------------

4. ORDER BY INJECTION RISK ❌

orderBy + order inserted raw.

Attacker could inject SQL via query params.

Fix:

Whitelist sortable fields.

Example:
allowedSortFields.includes(orderBy)

Never trust client sorting fields.

--------------------------------------------------

5. SUPERADMIN ACCESS MODEL UNCLEAR

organizationId = null → global access.

Danger:
accidental unrestricted queries.

Safer model:

explicit option:
allowGlobalAccess: true

Never infer global access implicitly.

--------------------------------------------------

6. TRANSACTION DESIGN DUPLICATION

You already built Transaction utility.

But repository also manages BEGIN/COMMIT.

Pick ONE model.

Recommended:
Only use withTransaction() helper.

Avoid dual transaction systems.

--------------------------------------------------

7. EXISTS() BUG

exists() calls:

count(conditions, client)

But count signature:
count(conditions, organizationId, client)

Organization filtering silently broken.

High-risk bug.

--------------------------------------------------

8. SELECT * PERFORMANCE ISSUE

findAll uses SELECT *.

Large tables →
wasted bandwidth + slower queries.

Prefer:

selectColumns option.

Enterprise rule:
never default to *.

--------------------------------------------------

9. MISSING SOFT DELETE SUPPORT

delete() performs hard delete.

SaaS systems usually require:

deleted_at column.

Soft deletes prevent data loss and audit issues.

--------------------------------------------------

10. NO QUERY OBSERVABILITY

Central query layer should log:

query time
table
operation
slow queries

This is ideal place for performance instrumentation.

2. INVENTORY CONSISTENCY RISK (VERY IMPORTANT)

You maintain:

quantity

Invariant must ALWAYS hold:

quantity =
  available_quantity +
  reserved_quantity +
  damaged_quantity +
  in_transit_quantity

Currently NOT enforced.

Add:

CHECK constraint OR trigger validation.

Otherwise silent drift happens.

Enterprise systems fail here.

--------------------------------------------------

3. MISSING SERIALIZABLE SAFETY MODEL

Concurrent orders can cause:

Order A reserves stock
Order B reserves stock

Your condition:

available_quantity >= X

is GOOD ✔
but still vulnerable under heavy concurrency without:

SELECT ... FOR UPDATE
or SERIALIZABLE isolation.

Recommendation:

Inventory operations must run inside explicit transaction
with row lock.

Inventory = locking domain.

--------------------------------------------------

4. STOCK MOVEMENT SHOULD BE MANDATORY

Currently:

reserveStock()
deductStock()
addStock()

do NOT automatically record movement.

This allows invisible mutations.

Enterprise rule:

NO inventory change without movement log.

Movement creation should be internal,
not optional.

Repository should enforce audit trail.
